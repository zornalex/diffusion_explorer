Wie funktionieren Diffusion Models?

Bevor wir trainieren: versteh, was passiert — vom Verrauschen bis zur U-Net-Architektur.

1a

Forward Process: Rauschen hinzufügen

Ein Diffusion Model beginnt damit, ein sauberes Bild x₀ schrittweise mit Gauß'schem Rauschen zu überlagern — über T = 400 Schritte. Am Ende ist das Originalbild vollständig in reinem Rauschen verschwunden.

Bewege den Schieberegler rechts: bei t = 0 siehst du das Original, bei t = 400 nur noch Rauschen.

Bei jedem Schritt t wird ein kleines bisschen Rauschen hinzuaddiert. Durch den Cosine-Schedule geschieht das am Anfang langsam (Struktur bleibt sichtbar) und am Ende schneller.

  1. Bestimme Rauschstärke ᾱ_t (sinkt von 1.0 auf ≈ 0)
  2. Sampele ε ~ 𝒩(0, I) (zufälliges Gauß-Rauschen)
  3. Berechne: xₜ = √ᾱₜ · x₀ + √(1 − ᾱₜ) · ε
xₜ = √ᾱₜ · x₀ + √(1 − ᾱₜ) · ε
ε ~ 𝒩(0, I), ᾱₜ = ∏ᵢ₌₁ᵗ αᵢ, αᵢ = 1 − βᵢ

Das Rauschen-Level ᾱₜ nimmt monoton ab — rechts angezeigt als "ᾱ_t". Das Modell lernt später diesen Prozess umzukehren.

1b

U-Net Architektur: das Modell

Das U-Net bekommt ein verrauschtes Bild xₜ und den Zeitschritt t und soll das addierte Rauschen ε̂ vorhersagen.

Die Architektur folgt dem klassischen U-Net-Prinzip: Encoder komprimiert die räumliche Auflösung (32→16→8 px), Bottleneck verarbeitet den komprimierten Kontext, Decoder expandiert wieder auf die Originalgröße (8→16→32 px).

Skip Connections (gestrichelte Linien) leiten Features direkt vom Encoder zum Decoder — dadurch gehen feine Details nicht verloren.

Klick auf einen Block im Diagramm rechts zeigt Details: Filter-Anzahl, Input/Output-Shape, Aktivierungsfunktion und den Zweck der Schicht.
  1. xₜ (32×32×1) + t wird konkateniert → 32×32×2 Input
  2. Encoder: 2× Conv32 → Pool → 2× Conv64 → Pool (Kontext extrahieren)
  3. Bottleneck: 2× Conv128 (globaler Kontext, 8×8)
  4. Decoder: UpSample + Concat(skip) → Conv64 → UpSample + Concat(skip) → Conv32
  5. Output: Conv1×1 → ε̂ (32×32×1, vorhergesagtes Rauschen)

~500k Parameter · Adam lr=0.0002 · Loss: MSE(ε̂, ε)

What are we doing?

Diffusion models learn a distribution — not just one image, but what a whole category of images looks like. Your few smileys are the dataset.

During training the model sees thousands of randomly noised versions of your drawings and learns one thing: how to predict the noise that was added.

At each training step, the model:

  1. Picks a random drawing from your dataset
  2. Picks a random timestep t (1–400)
  3. Adds exactly the right amount of noise for that t
  4. Asks the U-Net: "what noise did I add?"
  5. Adjusts the weights to get closer to the right answer
L = 𝔼[‖ε − ε_θ(xₜ, t)‖²]

Step 3

Conditioning — Generating on Demand

What if the model could generate a specific type of smiley? That's class conditioning — and it's the same idea behind "text-to-image".

From unconditional to conditional

So far the model learns all smileys together — it can generate some smiley. Conditioning adds a label to each training image so the model learns which type to generate.

The label is just a small number (0 = happy, 1 = neutral, 2 = sad), fed into the model alongside the timestep. At generation time you choose the label.

Stable Diffusion does the same thing — but instead of a class number, it uses a 768-dimensional CLIP text embedding as the condition. The architecture is identical in principle.

The U-Net receives an extra input — the class embedding c:

ε_θ(xₜ, t, c) where c = Embed(class_label)

The loss becomes:

L = 𝔼[‖ε − ε_θ(xₜ, t, c)‖²]

Classifier-free guidance (CFG) makes this even more powerful by training both a conditional and unconditional model and interpolating at inference time.

Beyond Images

Diffusion in Robotics

The same algorithm that generates smileys is now used to plan robot movements in real time.

Pixels → Actions

In image generation, diffusion creates a grid of pixel values. In robotics, it creates a sequence of joint angles or end-effector positions — a movement trajectory.

Diffusion Policy (Chi et al. 2023) trains a diffusion model on expert demonstrations. Given the current robot state, the model denoises a trajectory from noise → smooth motion plan.

Why diffusion? Robot movements are multimodal — there are often many valid ways to solve a task. Diffusion naturally captures this: different noise seeds produce different but valid trajectories.
  1. Observe robot state + camera image → encode as condition c
  2. Sample noise aᴛ ~ 𝒩(0,I) (action sequence)
  3. Run DDPM reverse process conditioned on c
  4. Execute the denoised action sequence a₀

Paper: diffusion-policy.cs.columbia.edu ↗

Flow Matching — A Faster Path

DDPM takes 400 noisy zigzag steps. Flow Matching goes straight. Same destination, much faster.

The intuition

DDPM gradually destroys data into noise through many small random steps. Reversing this requires many denoising steps too — typically 50–1000 for good quality.

Flow Matching instead defines a straight-line interpolation from noise to data and trains a vector field to follow that straight path. This means you need far fewer steps — often just 10–20.

Modern models using Flow Matching: Stable Diffusion 3, Flux, Sora, Meta's MovieGen.

Define a straight-line path between noise x₁ ~ 𝒩(0,I) and data x₀:

xₜ = (1−t)·x₁ + t·x₀, t ∈ [0,1]

The target vector field is constant and simple:

u_t(xₜ|x₀,x₁) = x₀ − x₁

Train a network v_θ(xₜ, t) to predict this field. At inference, integrate with an ODE solver (e.g. Euler, 20 steps).

Paper: Lipman et al. 2022 ↗
